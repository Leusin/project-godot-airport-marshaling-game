# 코딩 컨벤션 (설계 원칙)

리팩터링으로 정착시킨 규칙. 새 코드는 이걸 따른다. 핵심은 **"객체는 자기 책임 안의 것만 알고, 남을 명령하지 않는다."**

## 1. 책임 경계 — 자기 것만 안다

각 스크립트는 자기 책임 안의 것만 참조한다. 파일 맨 위 `##` 주석에 **무엇을 하고, 무엇을 안 하는지**를 한 줄로 밝힌다 (예: "판정에 관여하지 않고 …만 시각화").

## 2. 의존 방향 — 하위는 상위를 부르지 않는다 ★핵심

엔티티·컴포넌트·입력(하위)은 매니저·게임 규칙(상위)을 **호출하지 않는다.** 상위가 하위를 **구독**한다.

- ✅ `GameManager`가 `aircraft.hazard_hit`를 구독해 게임오버를 정함
- ❌ `Aircraft`가 `GameManager.trigger_game_over()`를 호출 ← 엔티티가 게임 규칙을 배달

**판별법**: "이 객체가 없어도 상대가 존재할 수 있나?" 엔티티는 매니저 없이도 존재해야 한다. 상대를 널가드(`if x != null`)로 버티고 있다면 방향이 틀렸다는 신호.

## 3. 사실 vs 해석 분리

물리적·도메인 **사실**("hit", "parked", "hand_signal")과 게임 **해석**("game over", "success", "확정 대기")을 나눈다.
엔티티는 사실(상태·시그널)만 노출하고, 그 의미는 판정자(GameManager 등)가 해석한다.

## 4. 뷰는 읽기만 — 로직을 구동하지 않는다

HUD·스프라이트는 상태를 **읽어 표시만** 한다. 게임 로직을 부르거나 상태를 바꾸지 않는다.
뷰가 여러 매니저의 상태를 읽는 건 최소화한다.
> 현재 예외: `marshaller_sprite`가 `GameManager.is_confirming_shutdown`을 읽어 엔진정지 포즈를 표시. 읽기 전용이라 허용하되 **이런 참조를 늘리지 않는다.**

## 5. 입력 분리

게임플레이 로직(FSM·Pawn)은 입력 장치(`SignalInput`/`MovementInput`)를 **직접 보지 않는다.**
입력은 도메인 값(`HandSignal`)이나 컨트롤러(`PlayerController`)를 거쳐 전달된다.

## 6. Controller / Pawn / 헬퍼

- **Pawn**(`Marshaller`/`Aircraft`): 명령받은 상태만 보유, 입력 무지.
- **조종자(Controller/brain)**(`PlayerController`/`AircraftFSM`): 입력·판단으로 Pawn을 조종. 조종자만 갈아끼우면 사람↔AI 교체.
- **헬퍼**(이동 계산 등 순수 로직): 씬 노드가 아니라 `RefCounted`. 소유 엔티티가 코드로 들고 호출한다.

## 7. 노드·타입 찾기

- 싱글턴 노드 → 그룹 조회(`SceneQuery.require_single`)
- 부모-자식 → 직접(`get_parent_node_3d()` 등)
- 유틸·도메인·헬퍼 → `class_name` 전역
- 오타 나기 쉬운 문자열(그룹명·입력 액션명)은 한 곳에 모은다(`GameGroups`, 각 스크립트의 `ACTION_*` 상수).
