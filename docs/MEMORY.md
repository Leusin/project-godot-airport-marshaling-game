## 프로젝트 목표

원하는 경험
- 조종이 아니라 유도하는 긴장감

익히고 싶은 것
- AI를 활용한 개발 워크플로우

가설
- AI는 보일러플레이트와 FSM은 잘한다.
- 조작감 튜닝은 사람이 해야 한다.
## 아이디어

### A 필수

- [x] 주차 정확도 등급 (B/A/S/SS): 완전포함 → 겹침비율(0.7) 게이트로 완화 + 확정 순간 위치·각도 오차로 채점. `AircraftCollision.parking_metrics()` + `ParkingGrade.evaluate()` + `SuccessHUD` 표시. 임계값은 실측 튜닝 대기

### B 폴리시

- [x] 시야 차폐(line-of-sight): `AircraftVision.can_see`가 반경→각도→시야선 순 판정. 비행기→대상 레이캐스트가 solid(장애물)에 걸리면 시야 밖. solid만 마스킹해 마샬러(hazard)·비행기 자신은 자동 제외(별도 exclude 불필요)
- [x] 마샬러 앞/뒤 방향: 비행기 바라봄 가정 → 비행기가 위면 등·아래면 정면. 좌/우 수신호 손잡이 교정 목적이라 `flip_h` 미러링만으로 해결(뒷모습 아트 불필요). Pawn `is_showing_back()` 사실 노출 + `set_facing_target` 주입, 뷰가 읽어 flip
- [ ] 레벨 로드맵을 버스 정류장식으로: 슬롯 사이 연결선 + 진행 표시 (현재 슬롯 나열로 충분해 보류)
- [x] 충돌 UI 효과: 히트스톱으로 구현 — 충돌 시 씬 정지 + 충돌 지점에 이펙트(HitEffectHUD, 월드→화면 투영) 0.5초 → 게임오버 HUD. 이펙트 이미지는 `assets/sprites/effects/impact.png` 넣으면 교체(없으면 임시 스파크)

## 회고 
(v0.3.0, 2026-06-28 ~ 2026-07-11)
첫 Godot 프로젝트였다. 엔진은 깊게 비교해서 선택한 것이 아니라, 호기심과 실행 속도 때문에 골랐다. 결과적으로 3일 만에 플레이 가능한 프로토타입이 나왔고, 13일 만에 레벨 캠페인을 갖춘 게임이 되었다.

### 역할 분담 기록

AI가 한 것:

- 코드 작성 사실상 전부. 게임 로직, 시그널 구조, HUD, 판정 규칙을 포함한다. 내가 직접 고친 것은 수치·씬 배치·문구 수준이다.
- 리팩터링 수행. 방향과 위반 지적은 사람이 했고, 실제 수정은 AI가 했다.
- 버그 추적과 수정.
- 레벨 디자인. 5개 레벨의 장애물·주차존·스폰 배치와 난이도 콘셉트를 AI가 설계했고, 나는 손대지 않았다. 튜닝은 사람 몫이라고 가정했던 영역인데 초안이 그대로 쓰였다.
- 테스트 작성과 유지.
- 문서 초안. DEVLOG 기록, 아키텍처 문서, 회고 초안.
- 비행기 3D 모델(Blender MCP), 아이콘 분할·배선(원본 생성은 Gemini).
- 화면 없는 검증. 게임을 창 없이(headless) 실행해 파서 에러·런타임 에러·테스트 통과를 확인.

내가 한 것:

- 기준 정의와 채택 판정. 구조를 받아들일지, 어디가 컨벤션 위반인지.
- 수치 튜닝. 등급 임계값, 카메라 위치, HUD 크기·위치·순서.
- 아트. 마샬러 빌보드, 주차장 마킹, 충돌 이펙트, 아이콘 수정.
- 시각 확인과 플레이테스트. 화면에 보이는 모든 것의 최종 판정.
- 조작감 판단. 명령 지연을 되돌린 결정.
- 방향 결정. 다음 작업 선정, 우선순위, 버전과 회고 시점.
- 회고의 최종 서술.

### Keep

**아키텍처**

- 아키텍처에 공을 들여 다이어그램까지 문서로 남겼다. 씬 레이어 구조는 중반(07-05)에 한 번 재편한 뒤로는 끝까지 유지됐다. 작업일의 약 40%를 리팩터링에 썼지만 수정은 모두 그 아래의 스크립트 의존 관계에서 일어났고, 이 층을 다시 설계할 일은 없었다.
- 구조를 먼저 정리한 것이 이후 기능 추가 비용을 낮췄다. 스폰 위치를 레벨 데이터로 분리해 둔 덕에, 레벨 캠페인 추가가 진행 매니저 하나와 배치 파일 다섯 개로 끝났다.

**문서**

- 결정과 교훈은 DEVLOG에 그날 기록했다. 세션이 끊겨도 기억을 더듬지 않고, 문서를 읽는 것만으로 바로 이어서 작업할 수 있었다. 이번 회고 역시 DEVLOG를 기반으로 작성했다.
- 문서 체제(CONVENTIONS, DEVLOG, ROADMAP, MEMORY)는 역할이 자연스럽게 분리되어 다음 프로젝트에서도 그대로 가져간다. 회고가 누적되어 MEMORY가 비대해질 때만 분리를 다시 검토한다.

### Problem

**리팩터링**

- 처음에는 AI가 코드를 빠르게 작성해 주면 리팩터링도 줄어들 것이라고 생각했다. 실제는 반대였다. 기능은 빠르게 붙었지만 코드의 방향도 빠르게 굳어 버렸고, 그 구조를 이해하고 다시 정리하는 시간이 계속 필요했다. 작업일 8일 중 3일, 약 40%를 리팩터링에 사용했다. 구조를 미뤘던 부분은 기능을 붙일수록 수정 비용이 더 커졌다.

**AI에게 맡긴 범위**

- 초기 가설("AI는 FSM을 잘한다")은 틀렸다. 상태 전이 자체는 잘 만들었지만 의존 관계를 난잡하게 엮었고, 결국 거의 수제로 다시 리팩터링했다.
- 처음에는 FSM이라는 유명한 패턴이니 구조도 자연스럽게 나올 것이라 생각했다. 하지만 패턴을 안다는 것과 프로젝트에 맞는 구조를 만드는 것은 전혀 다른 문제였다.
- AI는 패턴의 구현은 잘하지만, 그 패턴을 이 프로젝트의 의존 구조 어디에 어떻게 배치해야 하는지는 스스로 결정하지 못했다. 방향과 제약을 주면 구조 작업까지 해냈지만, 제약 없이 맡기면 스스로 좋은 구조를 잡지 못했다.

### Try

**AI 협업 규칙**

- 새 시스템을 맡길 때는 패턴 이름만 주지 않는다. 참조해도 되는 것과 참조하면 안 되는 것, 의존 방향까지 함께 명세한다.
- 조작감에 영향을 주는 값(입력 지연, 속도, 판정 타이밍)은 구조 개선이 목적이라도 플레이 확인 전에는 확정하지 않는다.

**운영 리듬**

- 컨벤션과 운영 문서를 프로젝트 첫날 만든다. 빈 저장소에서 시작하지 않고 이번 프로젝트의 보편 원칙(의존 방향, 사실/해석 분리, 뷰는 읽기만, 스폰과 배선은 상위가)을 시작점으로 복사한다.
- 기능 묶음이 끝날 때마다 정리 리뷰를 일정에 포함한다. 이번 사이클에서는 무적 게이트 잔류 버그가 기능 구현 중에는 보이지 않았고, 정리 리뷰에서 발견됐다.
- **기능 묶음이 완성되면 MINOR 버전을 올리고 태그를 찍는다. 사이클이 끝나면 회고를 작성한다.**

**다음 프로젝트**

- 다음 프로젝트로 가져가는 것은 코드 프레임워크가 아니라 운영 체계다. 문서 4종 틀, 보편 원칙 컨벤션, 협업 규칙(아트는 시안부터, 검증은 AI가 화면 없는 실행을 맡고 사람은 화면을 확인한다)을 그대로 시작점으로 사용한다.

### Unexpected

**문서의 역할**

- 컨벤션 문서의 역할이 예상과 달랐다. 규칙을 만들어도 AI는 여러 차례 위반했다. 대신 위반이 보이면 "이건 컨벤션 위반"이라고 말하는 것만으로 한 번의 수정으로 끝났다.
- 문서는 규칙을 강제하는 도구가 아니라, 사람과 AI가 같은 언어로 이야기하기 위한 인터페이스였다.

**AI의 역할 범위**

- AI의 역할 범위도 예상과 달랐다. 방향과 제약을 주면 시그널 구조, 책임 분리, 버그 추적, 레벨 초안까지 맡길 수 있었다.
- 반대로 비주얼은 거의 맡길 수 없었다. 마샬러 캐릭터는 AI 생성 결과를 폐기했고, 결국 직접 그린 빌보드를 사용했다. 이 선택이 게임의 최종 아트 스타일을 결정했다.
