- [x] 마샬러 앞/뒤 방향: 비행기 바라봄 가정 → 비행기가 위면 등·아래면 정면. 좌/우 수신호 손잡이 교정 목적이라 `flip_h` 미러링만으로 해결(뒷모습 아트 불필요). Pawn `is_showing_back()` 사실 노출 + `set_facing_target` 주입, 뷰가 읽어 flip

## 회고 (v0.3.0 사이클, 2026-06-28 ~ 07-11)

전제: 사실상 첫 Godot 프로젝트. 엔진 선택 이유는 호기심 + 빠른 실행 속도(Unity 2D보다 빨랐음).

### 결론

**AI 협업은 코드를 대신 써주는 게 아니라, 병목을 '작성'에서 '기준 정의와 판정'으로 옮긴다.**

- 속도는 진짜: 엔진 첫 경험에 MVP 3일, 캠페인 게임 13일
- 비용은 이동: 시간의 40%가 리팩터링. 규칙 문서화 후에도 위반은 계속 나옴 — 규칙의 실효는 방지가 아니라 "지적하면 한 턴에 고쳐지는 공용 언어"
- 사람에게 남은 일은 전부 판정: 튜닝 체감, 아트 채택/기각, 화면 확인, 방향 결정

한 문장: **AI는 개발자를 작성자에서 편집장으로 바꾼다.**

### Keep

- 구조 투자는 다음 기능의 선불 — 스폰 마커 리팩터 덕에 캠페인이 배치 작업만으로 완성
- 역할 분담: AI=구조·배선·판정 로직 / 사람=아트·튜닝·레이아웃
- DEVLOG 즉시 기록 — 세션이 끊겨도 다음 세션이 이어짐
- 플레이테스트 → 피드백 → 수정이 한 턴 루프
- 테스트 = 리팩터 안전망 + **AI가 혼자 닫을 수 있는 유일한 검증 루프** (자세한 평가는 아래)
- 검증 분업: AI=헤드리스(에러·테스트), 사람=시각·체감

### Problem

- 아트 역할 경계 침범 → 롤백 (규칙화: 비주얼은 시안·상의 먼저)
- 툴체인 마찰 반복: class_name 캐시 파서 에러(2회), MCP 실행 불안정, 워크트리 혼선
- 기능 몰아친 뒤에야 리뷰 → 무적 게이트 잔류 버그

### Unexpected

- **기대와 실제의 간극: 결과는 빠른데 내부가 읽기 어려워, 개발보다 리팩터링이 더 걸림.** 세션일 8일 중 3일+가 순수 구조 정리. 규칙을 넣어도 위반이 종종 나와 짧은 리팩터 주기가 상수 비용이 됨
- **AI 리소스 생성은 자산 종류로 갈림**: 정적 3D(비행기, Blender MCP) ⭕ / 2D 아이콘(Gemini) ⭕ / 캐릭터 외형+애니메이션(마샬러) ❌ → 손그림 빌보드로 전환, 이게 2.5D 아트 방향을 정함
- 프로토타입 "완성" 이후가 진짜 시작 — 폴리싱·구조 정리에서 게임이 게임다워짐
- AI의 영역이 예상보다 넓고(설계·버그 추적), 비주얼 감각은 예상보다 더 사람 영역

### 부록: 테스트 평가 (TDD 무지 상태에서 AI에 전임한 결과)

- 도움됨: 리팩터링 40% 물량을 매번 30/30 확인으로 담보. 실제 캐치 2건(멈칫 버그, 주석-코드 불일치)
- 한계: 임계값 오타·잔류 게이트·HUD 위치는 전부 테스트 밖(플레이·리뷰·눈)에서 잡힘
- 반례: 임계값 오타는 순수 함수라 테스트 한 줄이면 잡혔을 것 — 문제는 "무용"이 아니라 "안 깐 곳"
- 지침: 순수 규칙(FSM·도메인 함수)엔 깔고, 물리·UI·체감엔 안 깐다
