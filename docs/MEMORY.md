## 회고 (v0.3.0, 2026-06-28 ~ 2026-07-11)

첫 Godot 프로젝트였다. 엔진은 깊게 비교해서 선택한 것이 아니라, 호기심과 실행 속도 때문에 골랐다. 결과적으로 3일 만에 플레이 가능한 프로토타입이 나왔고, 13일 만에 레벨 캠페인을 갖춘 게임이 되었다.

### Keep

- 구조를 먼저 정리한 결정은 끝까지 도움이 됐다. 스폰 구조를 리팩터링한 뒤에는 캠페인 추가가 대부분 레벨을 배치하는 작업으로 끝났다. 이번 프로젝트에서는 새로운 기능보다 구조를 정리한 날이 이후 개발 속도를 더 크게 올렸다.
- 리팩터링할 때마다 테스트를 돌리는 것이 가장 큰 안전망이 되었다. TDD를 모르고 시작했고 테스트도 AI가 작성했지만, 실제로 테스트가 잡아낸 버그도 있었다.
- 컨벤션 문서가 있었기 때문에 "이건 컨벤션 위반"이라고 말하면 한 번의 수정으로 끝났다.

### Problem

- 처음에는 AI가 코드를 빠르게 써 주면 리팩터링도 줄어들 것이라고 생각했다. 실제는 반대였다. 새로운 기능은 빨리 붙었지만 코드의 방향이 빨리 굳어 버렸고, 그 구조를 이해하고 다시 정리하는 시간이 계속 필요했다. 프로젝트 기간 중 약 40%를 리팩터링에 사용했다. 구조를 미뤘던 부분은 기능을 붙일수록 수정 비용이 커졌다.
- 규칙을 만들면 AI가 규칙을 지킬 것이라고 생각했지만, 실제로는 규칙을 계속 위반했다.
- 초기 가설("AI는 FSM을 잘한다")과 달리 FSM을 제대로 만들지 못했다. FSM은 널리 알려진 패턴이라 당연히 잘 만들 것이라 생각했는데, 착각이었다. 상태 전이 로직 자체는 잘 동작했지만 의존 관계를 난잡하게 엉망으로 만들어서, 거의 수제로 리팩터링했다. 그대로 두면 새 기능이 들어올 때마다 구조를 이해하는 비용과 토큰 비용이 구현 비용보다 더 들 것 같았다. AI는 패턴의 로직은 알지만, 그것을 이 프로젝트의 의존 구조 어디에 어떻게 앉힐지는 모른다 — 패턴이 유명한 것과 구조가 깨끗한 것은 별개였다.
- 테스트가 모든 것을 해결하지는 못했다. 등급 임계값 오타는 플레이하면서 발견했고, HUD 문제는 눈으로 확인했고, 게임 감각은 결국 사람이 직접 플레이해야만 판단할 수 있었다. 테스트가 의미 없는 영역이 있었던 것이 아니라, 테스트를 만들지 않은 영역이 있었다.
- 가장 오래 붙잡았던 문제는 기능이 아니라 조작감이었다. 명령 지연 하나를 제거했더니 구조는 단순해졌지만 원래의 '쫄깃한' 느낌도 함께 사라졌다.

### Try

- 다음 프로젝트에서는 컨벤션과 운영 문서를 첫날부터 만든다.
- 기능을 일정량 구현할 때마다 리팩터링 시간을 일정에 포함한다.
- 순수 규칙은 테스트를 함께 작성하고, 게임 감각은 플레이로 확인한다.
- 코드보다 운영 방식을 재사용한다. 이번 프로젝트에서 다음 프로젝트로 가져갈 것은 프레임워크가 아니라, 문서와 협업 방식이다.

### Unexpected

- 컨벤션 문서의 역할이 예상과 달랐다. 위반을 막는 것이 아니라, 사람과 AI가 같은 언어로 이야기하게 만드는 것이었다.
- AI의 역할 범위도 예상과 달랐다. 처음에는 보일러플레이트나 FSM 정도를 잘 만들 것이라 생각했지만, 실제로는 시그널 구조, 책임 분리, 버그 추적까지 충분히 맡길 수 있었다. 반대로 비주얼은 거의 맡길 수 없었다. 마샬러 캐릭터는 AI 생성 결과를 폐기했고 결국 직접 그린 빌보드를 사용했다. 이 선택이 게임의 최종 아트 스타일을 결정했다.
- 좋은 구조가 항상 좋은 플레이 경험을 만드는 것은 아니었다. 구조는 리팩터링으로 되돌릴 수 있지만, 게임의 감각은 직접 플레이하면서만 찾을 수 있었다.
