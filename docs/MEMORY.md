## 회고 (v0.3.0 사이클, 2026-06-28 ~ 07-11)

전제: 사실상 첫 Godot 프로젝트였다. 엔진은 호기심과 실행 속도(더 단순한 Unity 2D 프로젝트보다 GDScript 3D 씬이 빨리 실행됨) 때문에 선택했다.

### 결론

**AI 협업은 코드를 대신 써주는 것이 아니라, 개발의 병목을 '작성'에서 '기준 정의와 판정'으로 옮긴다.**

- 속도 향상은 실제였다. 엔진을 처음 쓰면서도 MVP가 3일, 캠페인을 갖춘 게임이 13일에 나왔다.
- 비용은 사라지지 않고 이동했다. 세션 시간의 약 40%를 리팩터링에 썼고, 규칙을 문서화한 뒤에도 위반은 계속 나왔다. 규칙의 실제 효용은 위반 방지가 아니라, 위반을 지적하면 한 턴에 고쳐지게 하는 공용 언어 역할이었다.
- 사람에게 남은 일은 전부 판정이었다: 튜닝 값의 채택, 생성 아트의 채택과 기각, 화면의 시각 확인, 다음 작업의 결정.

요약하면, AI는 개발자를 작성자에서 편집장으로 바꾼다.

### Keep

- 구조 리팩터링이 다음 기능의 비용을 크게 줄였다. 스폰 마커 구조를 미리 정리해 둔 덕에 레벨 캠페인은 사실상 배치 작업만으로 완성됐다.
- 역할 분담이 명확할 때 빨랐다. AI가 구조·배선·판정 로직을 맡고, 사람이 아트·수치 튜닝·화면 배치를 맡았다.
- 결정과 교훈을 DEVLOG에 즉시 기록한 것이 세션 간 연속성을 만들었다.
- 플레이테스트에서 나온 피드백이 대부분 한 턴 안에 수정으로 이어졌다.
- 테스트가 리팩터링의 안전망이었고, AI가 사람 도움 없이 스스로 닫을 수 있는 유일한 검증 루프였다. 상세 평가는 아래 부록에 있다.
- 검증을 나눴다: AI는 헤드리스 실행과 테스트로 에러를 확인하고, 화면에 보이는 것은 사람이 눈으로 확인했다.

### Problem

- 아트 변경 의도를 AI가 구현 지시로 받아들여 임의로 바꿨다가 롤백했다. 이후 "비주얼은 시안과 상의가 먼저"로 규칙화했다.
- 툴체인 문제가 반복됐다: class_name 전역 캐시 미갱신으로 인한 파서 에러가 두 번, 에디터와 동시 실행 시 MCP 실행 불안정, 워크트리와 메인 폴더의 혼선.
- 기능을 몰아서 만든 뒤에야 리뷰해서, 무적 게이트가 남는 버그가 뒤늦게 발견됐다.

### Try

- 다음 프로젝트에서는 컨벤션 문서를 첫날 작성한다. 이번에는 중반 리팩터링 과정에서야 만들어졌다.
- 짧은 리팩터링 주기를 일정에 상수로 넣는다. 기능 묶음이 끝날 때마다 정리 리뷰를 한다.
- 도메인 함수를 새로 만들 때 테스트를 같이 만든다. 이번에 등급 임계값 오타가 테스트 없는 순수 함수에서 나왔다.
- 아트 협업 규칙을 유지한다: AI는 파일명 규칙과 배선만 준비하고, 그림은 사람이 그린다. 플레이스홀더를 쓸 때는 교체 전제임을 명시한다.
- 버전 규칙을 유지한다: 기능 묶음이 완성되면 MINOR를 올리고 태그를 찍고, 사이클이 끝나면 회고한다.
- 다음 프로젝트를 시작할 때 이 프로젝트에서 운영 체계를 추출한다. 코드 프레임워크는 장르가 바뀌면 맞지 않으므로 추출하지 않는다. 추출 대상: 문서 4종 틀(CONVENTIONS/DEVLOG/ROADMAP/MEMORY)과 운영 리듬, 게임 특화 사례를 뺀 보편 원칙판 컨벤션, 경량 테스트 하네스, 협업 규칙을 담은 CLAUDE.md.

### Unexpected

- 초기 기대와 실제가 가장 크게 갈린 지점: 결과물은 빨리 나오는데 내부 코드가 읽기 어려워서, 새 기능 개발보다 리팩터링에 더 많은 시간이 들었다. 세션일 8일 중 3일 이상이 순수 구조 정리였다. 규칙을 문서화한 뒤에도 위반이 종종 발생해, 짧은 주기의 리팩터링이 상수 비용이 됐다.
- AI 리소스 생성의 성패가 자산 종류에 따라 갈렸다. 비행기 3D 모델(Blender MCP)과 HUD 아이콘(Gemini 생성, Claude가 개별 파일로 분할)은 채택했다. 마샬러 캐릭터는 같은 방식으로 시도했지만 외형과 애니메이션이 모두 불완전해 폐기했고, 직접 그린 빌보드로 전환했다. 이 전환이 게임의 2.5D 아트 방향을 결정했다.
- 프로토타입 완성 이후가 진짜 시작이었다. 코어 루프를 만든 9일보다, 그 뒤의 폴리싱과 구조 정리 기간에 완성도가 올라갔다.
- AI가 잘하는 영역이 예상한 범위(보일러플레이트, FSM)보다 넓었다. 판정 규칙 분리, 시그널 설계, 버그 추적까지 해냈다. 반대로 비주얼 감각은 예상보다 더 사람의 영역이었다.

### 부록: 테스트 평가

TDD를 모르는 상태에서 테스트를 전적으로 AI에게 맡긴 결과다.

- 도움이 됐다. 세션 시간의 약 40%였던 리팩터링을 매번 전체 테스트 통과 확인으로 담보했다. 테스트가 직접 잡은 버그도 두 건 있다: RefCounted 전환 때의 멈칫 타이머 누락, 주석과 코드의 불일치.
- 한계도 명확했다. 등급 임계값 오타는 플레이테스트에서, 무적 게이트 잔류는 코드 리뷰에서, HUD 배치 문제는 눈으로 잡았다. 테스트가 잡은 것은 순수 로직뿐이다.
- 다만 임계값 오타는 순수 함수의 버그라서, 테스트를 깔아뒀다면 잡혔을 것이다. 경계는 "테스트가 소용없는 곳"이 아니라 "테스트를 깔지 않은 곳"이었다.
- 지침: 순수 규칙(FSM, 도메인 함수)에는 테스트를 만들고, 물리·UI·체감에는 만들지 않는다.

남은 항목: 게임 경험 목표("조종이 아니라 유도하는 긴장감")가 달성됐는지는 플레이하면서 평가해 채운다.
