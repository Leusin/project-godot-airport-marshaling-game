### Keep

- **구조 투자가 기능을 공짜로 만듦.** 엔티티 씬화+스폰 마커 리팩터(07-11) 덕에 레벨 캠페인이 사실상 배치 작업만으로 완성. 시그널 분리(매니저 상호 무지) 덕에 완료 화면을 별도 입력 핸들러 없이 기존 advance 경로에 얹음. "리팩터는 다음 기능의 선불"이 반복 확인됨.
- **역할 분담이 명확할 때 속도가 남.** AI=구조·배선·판정 로직, 사람=아트(아이콘·화살표·impact.png 전부 직접)·임계값 튜닝(등급 2배 완화)·레이아웃 감각(HUD 위치 반복 조정). 가설 "조작감 튜닝은 사람" 그대로 성립.
- **결정·교훈을 DEVLOG에 즉시 기록.** 세션이 끊겨도 다음 세션이 이어짐(스폰 마커 → 캠페인처럼 이전 기록이 다음 작업의 발판이 됨).
- **플레이테스트 → 즉시 피드백 → 즉시 수정 루프.** 등급 임계값 오타(A가 SS보다 빡빡), 카메라 여백, 키캡 위치 등이 한 턴 안에 잡힘.
- **엔진 선택(Godot)이 반복 루프와 맞물림.** 빠른 실행 속도가 "플레이테스트 → 피드백 → 수정" 루프의 토대. 엔진 경험이 거의 없어도 AI가 API·패턴을 채워줘 진입 장벽이 낮았음 — "새 엔진 배우기"에 AI 협업이 특히 유효.
- **경량 테스트 하네스가 리팩터 안전망 — 단, 순수 로직 섬에서만.** 매 변경마다 30/30 확인이 시간의 40%였던 리팩터링을 담보(테스트 없인 그 물량을 그 속도로 못 밈). 실제 캐치: RefCounted 전환의 멈칫 버그(07-09), 주석-코드 불일치(07-05). 반면 임계값 오타·잔류 게이트·HUD 위치는 전부 테스트 밖(플레이·리뷰·눈)에서 잡힘. 반례: `ParkingGrade.evaluate` 임계값 오타는 순수 함수라 테스트 한 줄이면 잡혔을 것 — 경계는 "테스트가 무용한 곳"이 아니라 "안 깐 곳"이었음. TDD가 아니라 회귀 안전망으로 썼고 그 용도로 충분히 값함. **AI 워크플로 관점의 특수 가치: 테스트는 AI가 혼자 닫을 수 있는 몇 안 되는 검증 루프**(시각 검증은 사람 필수).
- **검증 분업: AI는 헤드리스(콘솔 에러·테스트·구동)까지, 시각·체감(HUD 위치, 화살표 방향, 이펙트 위치)은 사람 눈으로.** AI가 화면을 못 보는 제약이 오히려 자연스러운 분업이 됨 — 시각 피드백 루프에는 사람이 반드시 낀다.

- **리뷰 없이 쌓다가 버그 잔류.** 히트스톱 도입 때 옛 무적 게이트가 남아 "일시정지 영영 안 풀림" 버그 — 기능 몰아친 뒤 정리 리뷰에서야 발견. 기능 묶음마다 정리 패스가 필요.

### Unexpected

- **초기 기대와 실제가 크게 달랐음: 결과는 빠른데 내부가 읽기 좋지 않아, 개발보다 리팩터링에 더 많은 시간이 듦.** DEVLOG가 이를 뒷받침 — MVP는 3일(06-28~07-01)에 나왔지만, 세션일 8일 중 3일+(07-05, 07-08~09)가 순수 구조 정리(인라인·전역조회·폴링 → Controller/Pawn·이벤트·판정 분리). 중반에 CONVENTIONS.md로 규칙을 문서화한 뒤에도 위반이 종종 발생해(07-11에도 뷰의 엔티티 직접 조회 §6 위반 2건, 히트스톱 잔류 게이트), **짧은 주기로 리팩터링을 반복하는 게 상수 비용**이 됨. 규칙은 위반을 없애주진 않고, 위반을 "지적하면 바로 잡히는" 상태로 만들어줌.
- **AI 리소스 생성의 성패가 자산 종류에 따라 갈림.** 비행기 3D 모델은 Blender MCP로 만들어 만족. 마샬러도 같은 방식으로 시도했으나 외형도 애니메이션도 불완전해 폐기 → 직접 그린 빌보드로 전환(이 결정이 게임의 2.5D 아트 방향을 정함). HUD 아이콘은 Gemini 생성이 만족스러웠고 Claude가 개별 파일로 잘라 배선. 요약: **정적 3D·2D 아이콘은 생성 AI가 됨, 캐릭터(외형+애니메이션)는 아직 사람 몫.** 생성 도구도 하나가 아니라 자산별로 골라 씀(Blender MCP / Gemini / 손그림).
- 프로토타입 "완성" 이후가 진짜 시작이었음 — 코어 루프(06-28~07-06, 9일)보다 그 뒤의 폴리싱·구조 정리(07-08~11)에서 게임이 게임다워짐.
- AI가 잘하는 영역이 예상(보일러플레이트·FSM)보다 넓었음: 판정 규칙 분리, 시그널 아키텍처, 버그 추적까지. 반대로 비주얼 감각은 예상보다 더 사람 영역이었음.

### 결론

초기 가설("AI는 보일러플레이트·FSM을 잘한다 / 튜닝은 사람")은 맞았지만 너무 좁았다. 이 프로젝트가 실제로 보여준 것은 역할의 구분이 아니라 **병목의 이동**이다.

> **AI 협업은 '코드를 대신 써주는 것'이 아니라, 개발의 병목을 '작성'에서 '기준 정의와 판정'으로 옮기는 것이다.**

1. **속도는 진짜다.** 엔진 첫 경험에 MVP 3일, 캠페인 게임 13일 — 작성 병목은 사라졌다.
2. **비용은 사라지지 않고 이동했다.** 시간의 40%가 리팩터링. 규칙을 문서화해도 위반은 계속 나오며, 규칙의 실효는 "방지"가 아니라 "지적하면 한 턴에 고쳐지는 공용 언어"다. 사람의 일이 코드 작성에서 **기준을 정의하고, 위반을 판정하고, 짧은 주기로 상환시키는 관리**로 바뀌었다.
3. **사람에게 남은 일은 전부 '판정'이다.** 임계값·카메라 체감(튜닝), 생성 아트 채택/기각(마샬러 3D 폐기 → 빌보드 전환), 화면 확인(시각), 다음 작업 선정(방향). AI가 시도하지 못해서가 아니라, **책임지고 결정하는 자리가 사람**이기 때문이다.

한 문장으로: **AI는 개발자를 작성자에서 편집장으로 바꾼다 — 빠르게 쓰는 건 AI지만, 무엇이 좋은지 정의하고 판정하는 속도가 새로운 병목이다.**
